# yStandard v5 開発計画

## リリース目標と前提

- リリース時期: 2026年夏
- 動作要件: WordPress 6.9+ / PHP 7.4+ 対応（開発環境は PHP 8.0+）
- 優先順位: **リリース > 完璧さ**。判定基準を満たしたら出荷する

## 設計方針

### コンセプト
- カスタマイズしやすさ国内 No.1 のテーマを目指す（設計上の指針。リリース判定基準ではない）
- theme.json を併用したハイブリッドクラシックテーマ（フルサイト編集テーマではない）
- テーマ側の挙動はシンプルに保つ。複雑な自動調整を増やさない

### ブロックテーマ機能との関係
- ブロックテーマ（theme.json）で設定可能な項目も、**なるべくカスタマイザー側で設定できる** ようにする
- theme.json はベースライン、カスタマイザーはユーザー向け設定 UI という位置付け

### CSS 方針
- セレクターの詳細度は**基本的に `0,1,0`**（クラス単独）になるよう設計する
  - 上書きのためだけに詳細度を上げない
  - ユーザー側での上書きカスタマイズを容易にするため
  - 詳細度を上げる必要がある場合はコメントで理由を明記
- **カスタマイザー設定で CSS に影響する項目は、直接 CSS ルールを出力せず CSS カスタムプロパティの値で上書きする**
  - 例: `:root { --ys-color-primary: #xxx; }` をカスタマイザー由来で出力し、既存 CSS は `var(--ys-color-primary)` を参照
  - 新規の設定項目を追加する場合も、まず CSS カスタムプロパティを介す設計を優先する

### 引き継ぐもの・捨てるもの
- **引き継ぐ**: カスタマイザー既存設定値（オプション名は不変更）
- **捨てる**: サイト背景色とコンテンツ背景色の差に応じた余白自動調整（全てユーザー設定に委ねる）
- **捨てる**: WordPress 6.8 以前対応のレガシーコード

## リリースブロッカー（夏リリースまでに必須完了）

### HTML/CSS 構造の完成
- ハイブリッドテーマ化（theme.json 導入）に伴う CSS 構造の再構築
- theme.json 由来の挙動の検証と統合（`global-padding`、`alignfull` など）
- ブロックで設定可能な項目もカスタマイザーで触れるように統合

### テンプレートパーツ構造の刷新
- `template-parts/parts/` に集約されている部品を意味別ディレクトリに分離
- 分離完了後に旧 `template-parts/parts/` を廃止（後から変えるとユーザー混乱のため今回で完遂）
- パーツを任意の位置で単独読み込みできるようにする
  - パーツファイル冒頭で必要情報を自前取得する形式を基本に
  - 既存のクラス経由パラメータ渡しも並行サポート（ystandard-toolbox 互換）

### カスタマイザー構造の再編
- カスタマイザーの階層を浅くする
- デザインパネルの廃止・分割
- 投稿タイプ別設定の新設（階層あり / なしで項目を自動判定）
- カラーパレット刷新（追加式・件数無制限）
- オプション名は変更せず既存設定値を引き継ぐ

### `ys_get_template_part` 廃止
- WordPress コアの `get_template_part` に統一
- 既存の呼び出し箇所（約 24 ファイル）をすべて置き換え
- スラグにファイルパスを渡す独自挙動は、コア関数で吸収できる形に再設計

## リリース後でよい項目

- クラスのシングルトン化 — エラーが出ない限り急がない。v5 リリース後に段階的に対応

## 実装方針詳細

### 投稿タイプ別設定の新設

#### 概要
- 旧 `inc/post/class-post-customizer.php`（投稿・固定ページ専用）を廃止済み
- 新 `inc/post-types/` に集約。公開投稿タイプ全般をサポート
- 「デザイン」パネル内の投稿ページ・固定ページ設定は削除

#### 投稿タイプ検出ロジック
- `inc/utils/class-post-type.php` の既存機能を活用
- `get_post_types(['public' => true])` で取得
- 除外: `attachment`、必要に応じて WP 標準投稿タイプ以外
- 階層判定: `is_post_type_hierarchical()`
- フォールバック: `get_fallback_post_type()` で `post` または `page` の設定を流用

#### 設定項目（階層なし: post like）
- レイアウト（1col / 2col）
- アイキャッチ画像表示タイプ（default / full）
- 本文エリア背景色
- 記事上部: アイキャッチ表示、投稿日・更新日、カテゴリー
- 記事下部: カテゴリー・タグ、著者情報、関連記事、次前記事

#### 設定項目（階層あり: page like）
- レイアウト（1col / 2col）
- アイキャッチ画像表示タイプ（default / full）
- 本文エリア背景色
- 記事上部: アイキャッチ表示、投稿日・更新日
- 記事下部: 著者情報

#### カスタマイザー配置
- 投稿タイプごとに独立したセクションを作成
- セクション名: 「[投稿タイプ名]ページ」
- カスタマイザートップレベル配置（デザインパネル外）
- 並び順: 標準投稿タイプ（post, page）→ カスタム投稿タイプ（アルファベット順）

#### 動的設定読み込み
- WordPress `init` アクション後に投稿タイプを検出
- 検出した投稿タイプに応じて動的にカスタマイザー設定を生成
- キャッシュ不要（投稿タイプは比較的固定）

### カスタマイザー「デザイン」パネル分割

- 旧「デザイン」パネルを廃止・再編

#### 「全体デザイン」パネル新設
- 「フォント・文字色」「リンクカラー」「サイト背景」を含む

#### 「メニュー・ナビゲーション」パネル新設
- 「グローバルナビゲーション」「ドロワーメニュー」を含む

#### トップレベルへ移動
- 「サイトヘッダー」「モバイル表示」「目次」「アーカイブページ」「フッター」「Copyright」をデザインパネル外へ出す
- カスタマイザー起動直後に表示されるメニューとして配置
- 表示順の調整を忘れない

### カラーパレット刷新
- 既存カラーパレットを上書きする方式を廃止
- ユーザーが色設定を追加していく方式に変更
- 追加件数の上限なし

### `ys_get_template_part` 廃止
- 実態は `\ystandard\Template::get_template_part`
- WP コアの `get_template_part` に一本化
- スラグにファイルパスを渡した際に該当ファイルを include する独自挙動は、コア関数で再現可能か要検証。再現できない場合は呼び出し側を書き換える

### `template-parts/parts` 整理
- `template-parts/parts/` の各ファイルを、意味カテゴリーごとのディレクトリに分割して `template-parts/` 直下へ配置
- 参照している箇所も合わせて更新
- 旧 `parts/` ディレクトリは廃止
